# TurboQuantDB — Configuration Guide

This guide covers every configuration dimension exposed by `Database.open()` and `db.search()`, how they interact, and which presets to use for common scenarios.

For full empirical numbers across 32 configs × 4 datasets (GloVe-200, arXiv-768, DBpedia-1536, DBpedia-3072) see [`_full_config_report.md`](../benchmarks/_full_config_report.md) (generated by `python benchmarks/full_config_bench.py --full`).

---

## Parameter Quick Reference

```python
db = Database.open(
    path,
    dimension,
    bits=4,                 # 2 or 4 — compression bits per subspace
    rerank=False,           # False (default) = MSE codes only, minimum disk
                            # True = store raw INT8 vectors for exact second-pass rescoring
    rerank_precision=None,  # None (→ "int8"), "int8"/"i8", "int4"/"i4", "f16", "f32", "disabled"
    fast_mode=True,         # True (default) = MSE-only (fastest ingest, minimum disk)
                            # False = MSE + QJL residual (+5–10 pp R@1 at d ≥ 1536, rerank=False)
    quantizer_type=None,    # None/"dense" (Haar QR) or "srht" (Hadamard)
    metric="ip",            # "ip" (inner product) or "l2" (Euclidean)
    seed=42,
)

results = db.search(
    query,
    top_k,
    rerank_factor=None,     # int multiplier for rerank oversampling; default 10 (brute) / 20 (ANN)
    _use_ann=None,          # None (auto), True (force HNSW), False (force brute)
)
```

## Dimension-by-Dimension Guide

### `bits` — 2 or 4

Controls the number of bits per codebook subspace (Lloyd-Max quantization).

| bits | Compressed size | Recall cost | When to use |
|------|----------------|-------------|-------------|
| 4    | ~0.5 bytes/dim | baseline    | Almost always |
| 2    | ~0.25 bytes/dim | −5 to −15 pp R@1 | Storage extremely constrained |

**Rule:** Use `bits=4`. The compressed size advantage of `bits=2` is modest (2× over `bits=4`) while the recall penalty is large. At `rerank=True`, the raw-vector storage dominates disk anyway, making the code size difference irrelevant.

---

### `fast_mode` — True (default, MSE-only) or False (MSE+QJL)

Controls whether 1-bit QJL residual codes are stored alongside the MSE codes.

| fast_mode | Ingest cost | Disk overhead | Recall effect |
|-----------|-------------|---------------|---------------|
| True **(default)** | baseline | 0 | baseline (all bits to MSE codebook) |
| False     | +20–40%     | +4–8 MB/100k vectors | d < 512: **−2 to −8 pp R@1** (noise) |
|           |             |               | d ≥ 1536: **+5–10 pp R@1** (signal accumulates) |

**Rule:**
- `d < 512` → `fast_mode=True` always. QJL projections at low d are too noisy and hurt recall.
- `d ≥ 1536` → `fast_mode=False` when `rerank=False`. QJL residuals add useful signal.
- `rerank=True` → `fast_mode` is secondary. Float precision re-scoring dominates regardless of residual quality.

---

### `rerank` + `rerank_precision` — exact rescoring

When `rerank=True`, raw vectors are stored on disk alongside compressed codes. During search, the top `rerank_factor × top_k` candidates from the quantized pass are re-scored with exact dot products against the stored raw vectors.

| rerank_precision | Storage per vector | Precision | When to use |
|------------------|--------------------|-----------|-------------|
| `"int8"` / `"i8"` **(default)** | d+4 bytes | ~0.5% error at unit norm | Default — 2× smaller than F16, same recall |
| `"int4"` / `"i4"` | ⌈d/2⌉+4 bytes | ~8% error | Tight storage budget; minor recall trade-off |
| `"f16"` | 2 bytes/dim | ~0.001 error | Maximum precision, storage not critical |
| `"f32"` | 4 bytes/dim | exact | Full precision; rarely needed |
| `"disabled"` / `"dequant"` | 0 | zero gain for IP | No raw storage; kept for legacy |

**Default:** `rerank_precision=None` → `"int8"` (INT8 per-vector-scaled, recommended).

Each slot stores a `f32` per-vector scale (4 bytes) plus one `i8` per dimension (INT8) or two nibbles per byte (INT4). For unit-normalized vectors, INT8 quantization error is ~8×10⁻⁵ per dimension — far below the score gaps between true nearest neighbors at any d.

**Disk impact at 100k vectors:**

| d    | rerank=False | int8 (default) | int4 | f16  | f32  |
|------|-------------|---------------|------|------|------|
| 200  | ~16 MB      | ~20 MB        | ~14 MB | ~40 MB | ~80 MB |
| 768  | ~48 MB      | ~74 MB        | ~40 MB | ~154 MB | ~307 MB |
| 1536 | ~96 MB      | ~149 MB       | ~79 MB | ~307 MB | ~614 MB |

**Recall impact (brute-force, bits=4):**

At d=200, R@1 improvement from rerank is modest (~+5–8 pp) because MSE codes already capture most of the signal. At d≥1536, the improvement is +15–25 pp R@1.

**`rerank_factor`** (search time, default 10 brute / 20 ANN):

The number of candidates pre-selected by the quantized pass before exact re-scoring. Higher factor = better recall at higher latency. Industry convention (Qdrant `oversampling`, LanceDB `refine_factor`):
- `rerank_factor=5`: fast, slight recall loss at top_k=1
- `rerank_factor=10`: default, good balance
- `rerank_factor=20–50`: maximum recall, use when latency is not critical

Default behavior note:
- The default oversampling is intentionally fixed (`10` brute-force / `20` ANN). It is not auto-adapted by dimension unless you pass `rerank_factor` explicitly.

```python
# High-recall, accepts 5× latency overhead
results = db.search(query, top_k=10, rerank_factor=50)

# Fast re-rank, accepts minor recall drop
results = db.search(query, top_k=10, rerank_factor=5)
```

---

### `quantizer_type` — `None`/`"dense"` vs `"srht"`

Selects the rotation applied before Lloyd-Max quantization.

| quantizer_type | Rotation | Ingest cost | Recall | When to use |
|----------------|----------|-------------|--------|-------------|
| `None` / `"dense"` | Haar QR | O(d²) | best | Default; d < 2048 |
| `"srht"` | SRHT (Hadamard) | O(d log d) | −1 to −3 pp | d ≥ 1536 with high throughput requirement |

**Rule:** Use `"dense"` (default). Switch to `"srht"` only when ingest throughput is the primary bottleneck at d ≥ 1536 (e.g., real-time embedding pipelines).

---

### `_use_ann` — brute-force vs HNSW

| `_use_ann` | When active | p50 latency | Recall penalty |
|-----------|-------------|-------------|----------------|
| `False` / `None` (auto, no index) | always | O(N) | none |
| `True` (requires `create_index()`) | N ≥ 10k | O(log N) | −2 to −8 pp R@1 |

The auto planner (`_use_ann=None`) enables HNSW when:
- An index exists
- N ≥ 10,000
- Unindexed delta ≤ 20% of corpus

Call `db.create_index()` after bulk load to build the HNSW graph.

## Recommended Presets

### Preset 1 — Max recall, high-dimensional (d ≥ 1536)

```python
db = Database.open(path, dimension=1536,
    bits=4, rerank=True, fast_mode=False,
    quantizer_type=None)  # dense, default; rerank_precision=None → int8
# ~149 MB for 100k vectors; R@1 ≈ 0.94–0.96 (DBpedia-1536)
```

### Preset 2 — Max recall, low-dimensional (d ≤ 768)

```python
db = Database.open(path, dimension=768,
    bits=4, rerank=True, fast_mode=True,
    quantizer_type=None)  # rerank_precision=None → int8
# ~74 MB for 100k vectors; R@1 ≈ 0.96–0.99 (arXiv-768)
```

### Preset 3 — Default (minimum disk, fastest ingest)

```python
db = Database.open(path, dimension=DIM, bits=4)
# rerank=False, fast_mode=True (defaults) — MSE codes only
# ~22 MB (d=200), ~48 MB (d=768), ~108 MB (d=1536) per 100k vectors
```

### Preset 4 — Minimum disk

```python
db = Database.open(path, dimension=DIM,
    bits=2, rerank=False, fast_mode=True)
# ~6–16 MB for 100k vectors; R@1 ≈ 0.51–0.84 (dataset-dependent)
```

### Preset 5 — Fast ingest, high-dimensional

```python
db = Database.open(path, dimension=1536,
    bits=4, rerank=False, fast_mode=True,
    quantizer_type="srht")
# 4,000–7,000 vps vs 2,000 vps for dense at d=1536
```

### Preset 6 — Low latency at large N

```python
db = Database.open(path, dimension=DIM,
    bits=4, rerank=True, fast_mode=True)
db.create_index(max_degree=32, ef_construction=200)
results = db.search(query, top_k=10, _use_ann=True)
# p50 < 10ms at N=100k; R@1 ~2–5 pp below brute-force
```

## Decision Flowchart

```
Start from defaults: bits=4, rerank=False, fast_mode=True, quantizer_type="dense"

Need better recall?
  YES → rerank=True (+5–25 pp R@1; adds INT8 raw vectors, ~2–4× more disk)
      → rerank_precision="int4" if storage is tight (~half disk vs int8)
      → rerank_precision="f16" for maximum precision

Is d ≥ 1536 AND rerank=False?
  YES → fast_mode=False adds QJL residual (+5–10 pp R@1)
  NO  → keep fast_mode=True (QJL noisy at d<512; redundant when rerank=True)

Is N > 100k or p50 < 10ms required?
  YES → create_index() + _use_ann=True (or None for auto)
  NO  → brute-force is fine

Is ingest throughput critical (streaming, d≥1536)?
  YES → quantizer_type="srht"
  NO  → quantizer_type="dense" (default)
```

## Storage Estimation

Approximate disk usage for 100k vectors:

```
compressed_codes_MB = N × ceil(d / (8 / bits)) / 1e6   # b=4: d/2 bytes, b=2: d/4 bytes
raw_int8_MB         = N × (d + 4) / 1e6                # rerank=True, int8 (default)
raw_int4_MB         = N × (ceil(d/2) + 4) / 1e6        # rerank=True, int4
raw_f16_MB          = N × d × 2 / 1e6                  # rerank=True, f16
raw_f32_MB          = N × d × 4 / 1e6                  # rerank=True, f32
hnsw_graph_MB       ≈ N × max_degree × 4 / 1e6         # default max_degree=32 → ~12 MB/100k
wal_overhead_MB     ≈ 5–20% of compressed_codes_MB
```

Example (d=1536, N=100k, bits=4, rerank=True, default int8):
- Compressed codes: 100k × 768 bytes = ~73 MB
- INT8 raw vectors: 100k × (1536 + 4) bytes = ~147 MB
- HNSW graph (optional): ~12 MB
- **Total: ~230–245 MB** (vs ~378–400 MB with F16)
